The problem

AI agents are great at clicking through web apps. They are terrible at logging in. Every authenticated workflow today either burns ten minutes asking the user to paste cookies, burns hours setting up a fragile sync extension, or just fails when the session expires.

Existing Chrome cookie sync tools are built for humans switching accounts between two laptops they both touch. They assume someone will click Merge or open Chrome periodically. agentcookie is built for the opposite workflow: continuous one-way replication from the machine you live in to the machine your AI agents act from. No human in the loop on the sink side. No browser open. No third-party storage.

How it works

SOURCE (laptop) SINK (Mac mini / cloud VM) agentcookie source ==== AES-GCM ====> agentcookie sink (launchd) read Chrome SQLite over HTTPS verify proto + sequence decrypt w/ Keychain on tailnet filter by allowlist filter by allowlist inject via Chrome DevTools wrap in envelope (or write SQLite) seal w/ peer key Tailscale tailnet, WireGuard, peer-to-peer

Pairing

X25519 ephemeral exchange salted with a one-time pairing code. Both sides derive an identical 32-byte symmetric key via HKDF-SHA256. MITM defense: an attacker who intercepts the channel without the code derives a different key.

Allowlist on both sides

The source filters cookies before sending. The sink filters again on receive, independently, against its own allowlist. The sink owner has the final say on what state lands in their Chrome, even if the paired source is compromised.

Live-Chrome injection

When Chrome is up with remote debugging, the sink writes cookies via Chrome DevTools Protocol (Storage.setCookies) so they are visible to running pages immediately. Falls back to SQLite write when Chrome is closed.

Replay defense + protocol versioning

Every payload carries a monotonic Sequence and a protocol version. The sink rejects out-of-order or duplicate payloads (replay defense) and rejects protocol mismatches loudly.

What's in v0.1

  • Unified agentcookie CLI: source, sink, pair, status, version. All support --json for agent callers.
  • macOS source + macOS sink. Chrome stable channel.
  • X25519 + HKDF pairing handshake with one-time code.
  • Per-pair 32-byte symmetric key stored at ~/.config/agentcookie/keys/<peer>.json mode 0600.
  • AES-256-GCM transport over Tailscale tailnet.
  • Live-Chrome cookie injection via Chrome DevTools Protocol; SQLite write fallback.
  • Wire protocol v1 with envelope, source hostname, monotonic Sequence.
  • Allowlist enforced on both source and sink, independently.
  • 42 unit tests across the crypto, transport, config, keystore, pairing, CDP, and protocol layers.
  • Apache 2.0 license. No telemetry. No hosted dependencies.

Threat model in one paragraph

agentcookie protects cookies in transit, defends against MITM during pairing, rejects wrong-secret payloads, and gives the sink owner final say via its own allowlist. It does not protect against root on either machine, Chrome compromise, OS user account compromise, or sites that bind sessions to device fingerprints. See docs/threat-model.md for the full read.

FAQ

Highlights below. Full FAQ at docs/faq.md.

Why not a Chrome extension?

Extensions assume a human will click Merge. agentcookie is for unattended sync to a machine you don't sit in front of.

Why Tailscale?

Zero third-party data plane, end-to-end WireGuard, free for personal use. Transport adapters for raw SSH and Cloudflare Tunnel are tracked for follow-up versions.

What about Linux sinks / Firefox / Safari?

v0.1 is macOS source + macOS sink, Chrome stable. Linux, Brave, Arc, and Edge are easy follow-ups. Firefox and Safari use different cookie stores entirely; bigger lifts.

Is this open source?

Apache 2.0. PRs welcome at github.com/mvanhorn/agentcookie.
