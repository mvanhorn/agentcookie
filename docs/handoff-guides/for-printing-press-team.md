# Hand-off: integrate agentcookie secrets-bus into printing-press

This document is written for a Claude Code session (or human contributor) working inside the `mvanhorn/cli-printing-press` repo. The goal is to make every PP CLI a first-class adopter of the agentcookie secrets bus.

## Context

agentcookie shipped v0.13 (the secrets-bus wire format) and v0.14 (the adoption standard). The adoption standard defines `agentcookie.toml`, a per-project manifest that declares participation in cross-machine secret sync. When an agentcookie user installs a PP CLI on their laptop, agentcookie auto-discovers the manifest and ships the CLI's auth tokens to their sink machine without any manual `agentcookie secret import-from`.

The agentcookie side is done. The PP side is two pieces:

1. Generator emits `agentcookie.toml` per CLI (declarative coupling).
2. Generator emits Go runtime code that reads from the bus at startup (functional coupling).

These two pieces are what this guide asks you to build.

## The standard (one-page summary)

- Manifest filename: `agentcookie.toml`, visible-by-default.
- Lives next to the CLI's other generated artifacts (alongside `.printing-press.json`).
- Plain TOML. Inert data. No exec at discovery time.
- Author-side helper library available at `github.com/mvanhorn/agentcookie/pkg/agentcookieadoption`.
- Reader library for the runtime side: `github.com/mvanhorn/agentcookie/pkg/agentcookiesecret`.
- Full spec: [docs/spec-agentcookie-secrets-bus-v2-adoption.md](../spec-agentcookie-secrets-bus-v2-adoption.md) in the agentcookie repo.

## What to build

### Part 1: emit `agentcookie.toml` per CLI

Hook the emit at the same lifecycle stage as `.printing-press.json` emit. The new file is byte-deterministic so regeneration is a no-op when inputs haven't changed.

Field mapping from existing PP metadata:

| `.printing-press.json` field | `agentcookie.toml` field |
|------------------------------|--------------------------|
| `cli_name` | `name` |
| `display_name` | `display_name` |
| `description` | `description` |
| (always) | `project_kind = "cli"` |
| `cli_name` (derived) | `[secrets.file].path = "~/.config/<cli_name>/config.toml"` |
| `auth_env_var_specs[].sensitive == true` | `[sync.keys].<NAME> = true` |
| `auth_env_var_specs[].sensitive == false` | `[sync.keys].<NAME> = false` |
| (always) | `[sync].default = false` (so `[sync.keys]` allowlist is explicit) |

Header comment in every generated file:
```
# Generated by printing-press <version> from <spec_path>.
# Do not edit; rerun generation to update.
# See docs/spec-agentcookie-secrets-bus-v2-adoption.md in agentcookie.
```

Idempotency rule: if an existing `agentcookie.toml` carries the marker `# agentcookie-manual-override`, skip emission for that CLI and log a stderr warning. Authors who hand-edit do so by removing the auto-generated header and adding the override marker.

Use the helper library to write the file:

```go
import "github.com/mvanhorn/agentcookie/pkg/agentcookieadoption"

m := &agentcookieadoption.Manifest{
    SchemaVersion: 2,
    Name:          meta.CLIName,
    DisplayName:   meta.DisplayName,
    Description:   meta.Description,
    ProjectKind:   "cli",
    Secrets: agentcookieadoption.Secrets{
        File: &agentcookieadoption.SecretsFile{
            Path: fmt.Sprintf("~/.config/%s/config.toml", meta.CLIName),
        },
    },
    Sync: agentcookieadoption.Sync{
        Default: false,
        Keys:    syncKeysFromSpecs(meta.AuthEnvVarSpecs),
    },
}
if err := agentcookieadoption.WriteTo(m, outputPath); err != nil {
    return fmt.Errorf("emit agentcookie.toml: %w", err)
}
```

### Part 2: emit bus-aware auth-load Go code

The generated CLI already has an auth-load file (call its existing path `internal/auth/load.go` or whatever your generator uses; the exact path is yours). Modify the template so the generated `Load()` function reads the bus first, then falls through to the existing file loader, then env vars. Pattern from `docs/runbook-secrets-bus-adoption.md` Pattern A.

Template-level changes:

```go
// At the top of the generated auth/load.go file:
import (
    "errors"
    "log"
    "github.com/mvanhorn/agentcookie/pkg/agentcookiesecret"
)

func Load() (*Config, error) {
    // (1) agentcookie secrets bus first.
    busEnv, err := agentcookiesecret.Load("{{ .CLIName }}")
    if err != nil && !errors.Is(err, agentcookiesecret.ErrInvalidCLIName) {
        log.Printf("agentcookiesecret: %v; falling back to local config", err)
    }

    // (2) Existing config-file load. Unchanged.
    cfg, err := loadConfigFile(...)
    if err != nil {
        return nil, err
    }

    // (3) Bus values override per-field. Generated per auth_env_var_specs.
    {{- range .AuthEnvVarSpecs }}
    if v := busEnv["{{ .Name }}"]; v != "" {
        cfg.{{ .GoFieldName }} = v
    }
    {{- end }}

    return cfg, nil
}
```

Notes:
- Bus-wins-over-env at the merge point. Spec section 11.2. Do not invert.
- For multi-field credentials (client_id + client_secret), emit one override block per `auth_env_var_specs` entry.
- The `GoFieldName` helper is yours; presumably the existing generator already converts UPPER_SNAKE to PascalCase.

Add `github.com/mvanhorn/agentcookie` to the generated CLI's `go.mod` via your existing dependency-emit step. The reader library is intentionally small; transitive deps are stdlib only (`pkg/agentcookiesecret` has the encoding/json + os.UserHomeDir kind of footprint).

### Part 3: regenerate every shipped CLI

Once Parts 1 and 2 are wired, the rollout is mechanical:

1. Pick tesla-pp-cli as the canary. Regenerate it.
2. Open a PR against the tesla-pp-cli repo with the generator output diff. Body links to this hand-off guide and the agentcookie v2 spec.
3. Land + verify end-to-end on Matt's machine: drop a known TESLA_AUTH_TOKEN in `~/.agentcookie/secrets/tesla-pp-cli/secrets.env` on the laptop, push to the sink, run `tesla-pp-cli auth status` on the sink, confirm the bus value was consumed.
4. Roll out the rest in batches of 5 with same-day verification.

Memory rule [[no-courtesy-prs-external-forks]]: all PP CLIs are user-owned; this rule does not block here. Memory rule [[evidence-on-every-pr]]: every PR includes the generator-output diff plus at least one verification log from the canary.

## Testing

The generator emit is covered by:

- A snapshot test: regenerate tesla against a fixture spec; assert byte-identical output.
- A round-trip test: emit a manifest, parse it back via `pkg/agentcookieadoption.Validate`, assert no error.
- An integration test (if your test harness can pull in agentcookie): emit a manifest, hand to the v2 parser at `github.com/mvanhorn/agentcookie/internal/secretsbus.ParseManifestV2`, assert no error.

The runtime emit is covered by:

- A template-snapshot test: render the auth-load template for a tesla spec; assert it compiles and the expected override blocks are present.
- A behavior test against a stub bus: populate a tmpdir bus with TESLA_AUTH_TOKEN, run the generated `Load()`, assert cfg.AuthToken == bus value.

## Non-goals (do not do)

- Do not add an agentcookie dependency to printing-press itself. Only generated CLIs depend on agentcookie's runtime reader; the generator depends only on `pkg/agentcookieadoption` (which is stdlib-only).
- Do not auto-import secrets from the user's machine. The CLI reads from wherever the manifest points at runtime; the generator never reads from `~/.config/<cli>/config.toml` itself.
- Do not modify the wire envelope or the v1 spec. Both are stable.

## When this lands

Adoption rollout per-CLI shows up in `agentcookie discover` on the user's machine as tier `explicit-manifest` (sourced from the per-CLI repo's generated `agentcookie.toml`). Pre-rollout CLIs continue to show up as tier `pp-cli-derived` via the in-memory auto-detect adapter the agentcookie source already runs against `~/printing-press/library/*/.printing-press.json`. Both tiers work; explicit is the goal.

## Questions, friction, or weird cases

The agentcookie repo issues are at `https://github.com/mvanhorn/agentcookie/issues`. Spec ambiguities, helper-library shortcomings, or "the field mapping doesn't fit my case" are valid issues there. The reference implementation at `internal/secretsbus/manifest_v2.go` is the tie-breaker on parser behavior.

That's the whole hand-off. The standard is small on purpose.
